1.	DESCRIÇÃO
O presente documento apresenta a estrutura dos campos, por funcionalidade, para utilização pelos serviços Quarkus. A descrição dos campos segue a nomenclatura disponível nos manuais nativos do SAT.
2.	ORIENTAÇÃO PARA DEFINIÇÃO DOS PARÂMETROS
a.	Abreviação da primeira coluna de “ENT / SAI” das tabelas de “Dados do Serviço”:
i.	E = Campo de Entrada
ii.	S = Campo de Saída
iii.	R = Campo de Entrada Obrigatório
iv.	R* = A pesquisa poderia ser feita por vários critérios, indicado em cada interface.
v.	Em Branco = A “tag” do campo é informada sem conteúdo.

Disponibiliza Informação de Mensagem Digital
1.	Descrição
Este caso de uso permite disponibilizar informações de mensagem digital, conforme a solicitação do sistema requerente (APP, ATC).
2.	Atores
Nesta seção estão listados os atores relacionados ao caso de uso. As descrições dos atores devem ser consultadas no Diagrama de Caso de Uso.
2.1.	APP
2.2.	ATC

4.	Fluxo de Eventos
Este caso de uso se inicia quando o sistema recebe a solicitação de consulta de informação de mensagem digital.
4.1.	Fluxo Básico – Disponibiliza Informação de Mensagem Digital
4.1.1.	O sistema recebe a solicitação do sistema requerente de mensagem digital.
Os dados recebidos do sistema requerente para solicitação de mensagem digital estão descritos em 8.1.
4.1.2.	O sistema valida os dados recebidos. FE11 FE2
O sistema verifica que:
- Os dados obrigatórios foram informados;
- Conta cartão identificada;
4.1.3.	O sistema verifica que a conta cartão informada está em atraso. FA1
Os dados identificados de conta cartão estão descritos 8.4.
4.1.4.	O sistema consulta o parâmetro de mensagem digital.
O sistema deve identificar a parametrização de Mensagem Digital conforme a funcionalidade “Mantém Parâmetro Mensagem Digital”.
4.1.5.	O sistema identifica a mensagem digital para envio conforme os dias de atraso da conta cartão.
Os dados identificados da mensagem digital cartão estão descritos 8.5.
4.1.6.	O sistema disponibiliza ao sistema requerente as informações da mensagem digital.
Os dados disponibilizados ao sistema requerente estão descritos em 8.2.
4.1.7.	Finaliza caso de uso.
4.2.	Fluxos Alternativos
FA1.	Conta Cartão não está em atraso
No passo 4.1.3 do fluxo básico, caso o sistema identifique que a conta cartão não está em atraso, ou seja, com 0 dias em atraso, o sistema deve executar os seguintes passos:
1.	O sistema disponibiliza ao sistema requerente as informações as informações da mensagem digital.
O sistema retorna os de dias de atraso igual a “0” e o título da mensagem e o texto da mensagem em branco.
Os dados disponibilizados ao sistema requerente estão descritos em 8.2.
2.	Finaliza caso de uso.
4.3.	Fluxos de Exceção
FE1.	Dados obrigatórios não informados
No passo 4.1.2 do fluxo básico, caso o sistema verifique que os dados obrigatórios não foram informados, o sistema deve executar os seguintes passos:
1.	O sistema disponibiliza ao sistema requerente informação de erro na operação solicitada.
O sistema deve disponibilizar a mensagem MA096.
As informações disponibilizadas estão descritas em 8.3.
2.	Finaliza caso de uso.
FE2.	Conta cartão inexistente
No passo 4.1.2 do fluxo básico, caso o sistema verifique que a conta cartão informada não existe, o sistema deve executar os seguintes passos:
1.	O sistema disponibiliza ao sistema requerente informação de erro na operação solicitada.
O sistema disponibiliza mensagem. MA055
As informações disponibilizadas estão descritas em 8.3.
2.	Finaliza caso de uso.

Especificação Suplementar
1	Objetivo
O objetivo deste documento é definir os requisitos que não são capturados imediatamente nos casos de uso do modelo de casos de uso, ou seja, requisitos não funcionais e condicionantes aplicáveis ao projeto SIPCS como um todo. Este documento em conjunto com o Modelo de Casos de Uso abrange todos os requisitos do sistema.
Termos e abreviaturas específicos podem ser encontrados no Glossário do projeto.
2	Requisitos Não Funcionais
2.1	Usabilidade
2.1.1	Acessibilidade
2.1.1.1	O acesso às funcionalidades deve ser feito por menu, que deve ser apresentado na parte superior da tela, utilizando o conceito de "abas".
2.1.1.2	O acesso a cada funcionalidade deve ser feito com até no máximo 05 (cinco) cliques.
2.1.1.3	O sistema deve ser dotado de recursos como lupa, visando atender aos usuários portadores de deficiência visual.
2.1.1.4	O sistema deve estar totalmente aderente aos padrões impostos pelo W3C no que se refere à sintaxe e semântica da linguagem HTML. O sistema deverá passar pelos testes de acessibilidade do W3C.
2.1.1.5	O sistema deve exibir, no canto superior esquerdo da tela, a navegação do usuário, de forma a permitir a visualização de qual a tela e qual o módulo o usuário se encontra.
2.1.2	Coerência da Interface de Usuário
2.1.2.1	A interface gráfica do sistema deve seguir a interface padrão e compatível com os demais sistemas corporativos da CAIXA.
2.1.3	Estética
2.1.3.1	O sistema deve seguir o padrão de GUI definido no "Guia de Usabilidade para Desenvolvimento de Sistemas". Este guia define padrões para tamanho, espaçamento e cores de fontes.
2.2	Confiabilidade
2.2.1	Capacidade de Recuperação
2.2.1.1	O sistema deve possuir mecanismo de tolerância à falha que possibilite se mantiver operante após a ocorrência de problemas de ambiente ou por operações anormais realizadas pelos usuários:
	Tolerância a falhas no sistema de armazenamento, supridas pelo espelhamento de hardware / software em site de contingência, e pela replicação de bases por Oracle RAC;
	Tolerância a falhas no sistema de processamento de transações, supridas pelo duplo módulo autorizador, previamente balanceado na recepção de transações provenientes dos servidores das bandeiras;
	Tolerância a falhas em algumas das máquinas, supridas pelo duplo módulo autorizador.
2.2.1.2	O sistema deve possibilitar que uma transação seja desfeita (rollback), caso ocorram problemas relacionados à infraestrutura (timeout da rede, servidor, banco, etc).
2.2.2	Disponibilidade
2.2.2.1	O sistema deve estar disponível 24 horas por dia, 07 dias por semana.
2.2.3	Precisão
2.2.3.1	O sistema deve ser preciso no registro das transações e em todos os cálculos realizados, com duas casas decimais após a vírgula para determinar o correto resultado.
Para plataforma Alta: Dependendo da operação realizada entre os campos origem e destino, sendo os tamanhos dos campos divergentes, ocorre o arredondamento de casas decimais, conforme regra negocial definida.
2.2.3.2	Para garantir a disponibilidade dos dados o sistema deve possuir mecanismos de prevenção que monitorem o tamanho das tabelas de histórico com o objetivo de arquivar as informações o que impedirá que as tabelas de histórico cheguem ao limite de armazenamento e causem a indisponibilidade dos dados.
2.3	Desempenho
2.3.1	Tempo de Ativação (Start-up Time)
2.3.1.1	O sistema deve ser ativado no servidor de aplicações e de maneira individual em no máximo 3 minutos.
2.3.2	Tempo de Desativação (Shutdown Time)
2.3.2.1	O sistema deve ser desativado no servidor de aplicações e de forma individual em no máximo 3 minutos.
2.3.3	Tempo de Recuperação
2.3.3.1	Em caso de ocorrência de falha que derrube o sistema, o tempo médio de recuperação deve ser de 30 (trinta) minutos. O tempo máximo de recuperação deve ser de até 01 (uma) hora.
2.3.4	Tempo de Resposta
2.3.4.1	O tempo máximo de resposta para operações de inclusão, alteração e exclusão não deve ultrapassar 02 (dois) segundos.
2.3.4.2	Para operações envolvendo listagens e/ou relatórios, o tempo máximo de resposta não deve ultrapassar 04 (quatro) segundos.
2.3.4.3	Para operações envolvendo apresentação de gráficos, o tempo máximo de resposta não deve ultrapassar 10 (dez) segundos.
2.3.4.4	Para operações envolvendo autorizações de transações, o tempo máximo de resposta não deve ultrapassar 07 (sete) segundos.
2.3.4.5	O tempo de atualização das informações do painel de monitoração da autorização deve ser de, no máximo, a cada 1 minuto.
2.3.5	Vazão (Throughput)
2.3.5.1	Para as operações de transferência de informações entre os sistemas, a taxa de transferência deve ser de, no mínimo, 1.500 (mil e quinhentos) registros por segundo.
2.3.5.2	O sistema deve suportar ao menos 3.000 (três mil) acessos simultâneos de usuários.
2.3.6	Tempo limite para processamento de Jobs relacionados com funcionalidades e relatórios contábeis na rotina diária
2.3.6.1	Na parte final da diária são executados uma sequencia de Jobs relacionados com funcionalidades e relatórios contábeis. A execução desses processos é pesada devido à quantidade e tamanho dos registros processados. Esta situação ocasiona demora no fechamento da rotina e a na disponibilização de relatórios que dependem da execução dos processos. Para reduzir o tempo de finalização da diária e melhoria no desempenho da mesma, o sistema deve realizar uma redistribuição dos processos de SORT que geram os arquivos de trabalho que servem como origem para conformar os relatórios associados nesses processos contábeis.

2.4.9.2.1	O SIPCS deve verificar se há caracteres com acentuação (ou seja, caracteres iguais a "Ã" ou "Õ" ou "É” ou “Ç”, etc.) e em caso afirmativo substituí-lo pelo próprio caractere sem o acento.

4	Requisitos Condicionantes
4.1	Design
4.1.1	Ajuda
4.1.1.1	O sistema deve disponibilizar “Ajuda” geral e específica para todas as funcionalidades do mesmo.
4.1.2	Auditoria
4.1.2.1	O sistema deve gerar log de transações executadas e armazenar tais informações com a identificação do terminal, hora/data e usuário que executou a tarefa.
4.1.2.2	Deverão ser registrados em LOG os eventos da execução de todas as etapas do processo de instalação e atualização, tanto em sucesso quanto em falha.
4.1.3	Autenticação
4.1.3.1	A autenticação de usuários no sistema:
SSO
O Usuário solicita a requisição de acesso;
Usuário informa login e senha;
O SSO valida as credenciais e retorna o perfil do usuário;
O SSO retorna token contendo o usuário e perfil.
Login Único
O Login Único descriptografa o token e recupera o usuário e perfil;
O Login Único retorna os sistemas que o perfil tem acesso;
O Login Único gera o cookie criptografado do token e coloca no navegador.
SIPCS
O SIPCS descriptografa o cookie;
O SIPCS pega o token;
O SIPCS descriptografa o token e recupera o usuário e perfil;
O SIPCS pega o usuário e perfil e coloca na sessão;
O SIPCS monta o menu do sistema com base no perfil do usuário.
Ao efetuar nova requisição, valida se o cookie e o token estão ativos:
Sim - continua a operação;
Não - Efetua login.
4.1.3.2	Todos os acessos ao WebService serão validados pelo Open LDAP/LDAP para verificação do usuário e senha, retornando o respectivo perfil de acesso. Todos os serviços estão contemplados por um único WebService. A distinção entre os diferentes serviços é realizada dentro dos parâmetros que são passados para o WebService. Especificamente, estes parâmetros terão a identificação da entidade solicitante, o usuário e a senha, o serviço correspondente no módulo SAT, o tipo do serviço (leitura e/ou gravação) e os dados da transação.
4.1.3.3	Solicitação de Token (DATAPREV)
	Para a efetiva execução da operação de inclusão de desconto de RMC é necessário que seja feita autenticação e conectividade no gateway de API da DATAPREV.
	Deve ser solicitado um token (com validade de 1 hora), que deve ser utilizado junto com o certificado digital para fazer a autenticação, conforme usuário e senha disponibilizados pela DATAPREV.
	O período de expiração de um token é de 1 hora:
o	Antes deste período de expiração, pode-se fazer a renovação do token; ou
o	Após este período, deve-se obter um novo token.
o	Exemplos:
Atual access_token":"xxxxxxxx-xxxx-xxxx-xxxxxxxxxxxxxxxxx"
GW: https://hisdf.dataprev.gov.br (​https:​/​​/​hisdf.dataprev.gov.br" \o "https:​/​​/​hisdf.dataprev.gov.br​/​" \t "_blank​)
Para revogar o atual access token:
curl --cacert [cadeiacacert.pem] --cert [certificado.pem:senha] -d "token=[xxx]" -H "Authorization: Basic [xxx]" -H "Content-Type: application/x-www-form-urlencoded"
https://hisdf.dataprev.gov.br (​https:​/​​/​hisdf.dataprev.gov.br" \o "https:​/​​/​hisdf.dataprev.gov.br​/​" \t "_blank​)/revoke
Para obter novo token:
curl --cacert [cadeiacacert.pem] --cert [certificado.pem:senha] -d "grant_type=password&username=[username]&password=[passw ord]" -H "Authorization: Basic [xxx]"
https://hisdf.dataprev.gov.br (​https:​/​​/​hisdf.dataprev.gov.br" \o "https:​/​​/​hisdf.dataprev.gov.br​/​" \t "_blank​)/oauth2/token
	O formato dos serviços é REST/JSON, padrão OpenAPI 2.0;

• CN = O common name é composto pelo host+domínio internet registrado pela IF.
• OU = (Organization Unit) Nome da Instituição
• OU = nnnnnnnn (onde nnnnnnnn é o número base do CNPJ)
• OU = YYY (Onde YYY é o código do CBC – quando é uma IF ou "DTP" – quando é a Dataprev)
• O = ICP-Brasil (campo preenchido automaticamente pela AC emissora do certificado)
• C = BR (campo preenchido automaticamente pela AC emissora do certificado)
Obs.:

• Os certificados criados nos padrões anteriores serão aceitos normalmente.
4)	Poderão ser utilizados opcionalmente os campos "L" (localidade) e/ou "S" (estado).
5)	É vedado o reuso das chaves públicas utilizadas. Ao solicitar a emissão de um novo certificado para uso no e-Consignado, é imperativo gerar uma nova chave pública. Certificados emitidos para ambientes diferentes (produção e homologação) devem conter chaves públicas diferentes.

4.3.1.2	O sistema deve utilizar o padrão estabelecido no normativo TE080 para definição correta da nomenclatura de arquivos do sistema. O sistema deve considerar, entre outras, as seguintes regras:
Regra de Formação de DSNAME de arquivos e leiautes de relatório, conforme TE080.
ABCD = Qualificadores obrigatórios para os dsnames dos arquivos VSAM e seqüencial, gerenciados pelo  Storage Management Subsystem – SMS. Onde:
	A - Deve ser preenchido com os valores abaixo:
o	O = Aplicação Online.
o	B = Aplicação Batch.
	B - Código alfanumérico que define os critérios de gerenciamento de espaço para migração/deleção, onde:
o	A = 0 dia em L0;
o	B = 1 dia em L0;
o	C = 0 dia em L0 / 1 dia em L1 / 6 dias em L2;
o	D = 7 dias em L0 / 28 dias em L2;
o	E = 2 dias em L0;
o	F = 1 dia em L0 / 30 dias em L1;
o	G = 2 dias em L0 / 40 dias em L2.
Considerando:
o	L0 - Disco que o arquivo foi gerado;
o	L1 - Arquivo migrado para disco;
o	L2 - Arquivo migrado para fita.
	C - Código alfanumérico que define os critérios de retenção (tempo/versões) para gerenciamento de backup, onde:
o	W - 7 dias;
o	Q - 15 dias;
o	M - 35 dias;
o	K - 45 dias;
o	N - 60 dias;
o	T - 90 dias;
o	S - 180 dias;
o	A - 1 ano;
o	B - 2 anos;
o	C - 3 anos;
o	D - 5 anos;
o	E – Eventual;
o	Z - 13 versões;
o	X - 0 dia.
	D = Código numérico que define a necessidade e a quantidade de cópias do arquivo, onde:
o	0 - Sem cópia;
o	1 - Cópia para “backup” interno;
o	2 - Cópia para “backup” interno e externo.
Da composição da nomenclatura dos arquivos enviados ao SIGAP:
	D = indica o tipo de tecnologia (Dual);
	1 = código numérico que indica a bandeira Visa;
	2 = código numérico que indica a bandeira Master;
	3 = código numérico que indica a bandeira Elo;
	21 = código numérico que indica embossadora “Thomas Greg”;
	26 = código numérico que indica embossadora “Valid”;
	CN = código que define o valor do Tipo de Movimento (Conta Nova).
	RE = código que define o valor do Tipo de Movimento (Reemissão).
	RV = código que define o valor do Tipo de Movimento (Renovação).
	01 = sequencial de quebra: código numérico de controle sequencial para quebra do arquivo a cada 30.000 registros.

Não se aplica.
4.3.4	Linguagens de Implementação
4.3.4.1	Deve-se utilizar EJB (Enterprise Java Beans) para implementação dos objetos de controle;
4.3.4.2	Deve-se utilizar JNI (Java Native Interface) para interface com os programas COBOL;
4.3.4.3	Deve-se utilizar JSP (Java Server Pages) para confecção das páginas web;
4.3.4.4	Deve-se utilizar JAVA na implementação das classes que servirão como interface entre as páginas JSP e programas COBOL;
4.3.4.5	Deve-se utilizar AJAX para o Front-end;
4.3.4.6	Deve-se utilizar COBOL para a implementação das classes de negócio; Além disso, a utilização de COBOL Mainframe para as bases DB2 e COBOL Microfocus para as bases Oracle;
4.3.4.7	Deve-se utilizar C em versão Unix para as interações com o S.O.;
4.3.4.8	Deve- se utilizar a linguagem SQL para o SGBD do sistema.
4.3.5	Suporte a Plataformas
4.3.5.1	O sistema deve suportar acesso por meio dos navegadores Mozilla Firefox 2.0 ou superiores e Internet Explorer 6.0 SP2 ou superiores.
4.3.5.2	O sistema deve prover integração com programas COBOL/CICS. 
4.3.5.3	O sistema deve ser desenvolvido baseado em um framework que garanta sua portabilidade nas múltiplas plataformas operacionais utilizadas na CAIXA (SUN/SOLARIS, IBM, JBoss Aplication Server, Java, Windows, Unix, etc).
4.4	Interface
4.4.1	Formato de Interface
4.4.1.1	O protocolo sobre o qual o sistema deve executar deve ser o TCP/IP.
4.4.1.2	O tráfego de informações com os sistemas externos deve ser feito em formato binário ou XML, salvo para comunicação com programas COBOL.
4.4.1.3	O sistema deve executar interfaces de comunicação, transmissão e transferência de dados em conformidade com a norma NBR ISO/IEC 27002:2007.
